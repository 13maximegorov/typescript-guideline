# HTML/CSS Style Guide

Оригинальное руководство по HTML и CSS: [https://google.github.io/styleguide/htmlcssguide.html](https://google.github.io/styleguide/htmlcssguide.html).

## Введение

Это руководство описывает правила для оформления и форматирования HTML и CSS кода. Его цель — повысить качество кода и облегчить совместную работу и поддержку инфраструктуры.

Это относится к рабочим версиям файлов использующих HTML, CSS и GSS

Разрешается использовать любые инструменты для минификации компиляции или обфускации кода, при условии, что общее качество кода будет сохранено.

### Общие правила оформления

#### Протокол

**По возможности используйте HTTPS для встроенных ресурсов.**

Всегда используйте HTTPS (`https:`) для изображений и других медиафайлов, таблиц стилей и скриптов, если только соответствующие файлы не доступны по HTTPS.

```html
<!-- ❌ Не рекомендуется: отсутсвует протокол -->
<script src="//ajax.googleapis.com/ajax/libs/jquery/3.4.0/jquery.min.js"></script>

<!-- ❌ Не рекомендуется: используется HTTP -->
<script src="http://ajax.googleapis.com/ajax/libs/jquery/3.4.0/jquery.min.js"></script>
```

```html
<!-- ✅ Рекомендуется -->
<script src="https://ajax.googleapis.com/ajax/libs/jquery/3.4.0/jquery.min.js"></script>
```

```css
/* ❌ Не рекомендуется: отсутсвует протокол */
@import '//fonts.googleapis.com/css?family=Open+Sans';

/* ❌ Не рекомендуется: используется HTTP */
@import 'http://fonts.googleapis.com/css?family=Open+Sans';
```

```css
/* ✅ Рекомендуется */
@import 'https://fonts.googleapis.com/css?family=Open+Sans';
```

### Общие правила форматирования

#### Отступы

**Всегда используйте для отступа два пробела.**

Не используйте табуляцию и не смешивайте табуляцию с пробелами.

```html
<!-- ✅ Рекомендуется -->
<ul>
  <li>Пять</li>
  <li>Погулять</li>
</ul>
```

```css
/* ✅ Рекомендуется */
.example {
  color: blue;
}
```

#### Регистр

**Всегда пишите в нижнем регистре.**

Весь код должен быть написан в нижнем регистре. Это относится к названиям элементов, названиям атрибутов, значениям атрибутов (кроме `text/CDATA`), CSS, свойствам и их значениям (кроме текста).

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется -->
<a href="/">Home</a>
```

```html
<!-- ✅ Рекомендуется -->
<img
  src="google.png"
  alt="Google"
/>
```

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется */
.test {
  color: #E5E5E5;
}
```

```css
/* ✅ Рекомендуется */
.test {
  color: #e5e5e5;
}
```

#### Пробелы в конце строки

**Убирайте пробелы в конце строки.**

Пробелы в конце строк не обязательны и усложняют использование diff.

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется -->
<p>Что?_
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
<p>Вот так.
```

### Общие мета правила

#### Кодировка

**Используйте UTF-8 (без BOM).**

Убедитесь, что ваш редактор использует кодировку UTF-8 без метки порядка байтов (BOM).

Указывайте кодировку в HTML шаблонах и документах с помощью `<meta charset="utf-8">`. Опускайте кодировку для сss-файлов: для них UTF-8 задана по умолчанию.

(Вы можете узнать больше о кодировках, и о том, как их использовать, по этой ссылке: [Handling character encodings in HTML and CSS](https://www.w3.org/International/tutorials/tutorial-char-enc/).)

#### Комментарии

**По возможности поясняйте свой код, где это необходимо.**

Используйте комментарии, чтобы пояснить свой код: что он делает, за что отвечает, и почему используется выбранное решение.

(Этот пункт не обязателен, потому что нет смысла ожидать, что код всегда будет хорошо задокументирован. Полезность комментирования зависит от сложности проекта и может различаться для HTML и CSS кода.)

#### Задачи

**Отмечайте задачи для списка дел с помощью `TODO`.**

Отмечайте задачи с помощью ключевого слова `TODO`. не используйте другие часто встречающиеся форматы, такие как `@@`.

Заключайте контакты (имя пользователя или список адресатов) в круглые скобки: `TODO(контакт)`.

Описывайте задачу после двоеточия, например: `TODO: Задача`.

{/* prettier-ignore */}
```html
{# TODO(Петя Пупкин): Разобраться с центрированием #}
<center>Test</center>
```

{/* prettier-ignore */}
```html
<!-- TODO: Убрать необязательные теги -->
<ul>
  <li>Масло</li>
  <li>Маргарин</li>
</ul>
```

## HTML

### Правила оформления HTML

#### Тип документа

**Используйте HTML5.**

HTML5 (HTML синтаксис) рекомендуется для всех HTML-документов: `<!DOCTYPE html>`.

(Рекомендуется использовать HTML с типом контента `text/html`. Не используйте XHTML, так как `application/xhtml+xml` хуже поддерживается браузерами и ограничивает возможность оптимизации. )

#### Валидность HTML

**По возможности используйте валидный HTMl.**

Используйте валидный HTML код, кроме случаев, когда использование не позволяет достичь размера файла, необходимого для нужного уровня производительности.

Используйте такие инструменты как [W3C HTML validator](https://validator.w3.org/nu/) чтобы проверить валидность кода.

Валидность — это важное и при этом измеряемое качество кода. Написание валидного HTML способствует изучению технических требований и ограничений и обеспечивает правильное использование HTML.

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется -->
<title>Проверка</title>
<article>Просто проверка
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
<!DOCTYPE html>
  <meta charset="utf-8">
  <title>Проверка</title>
  <article>Просто проверка.</article>
```

#### Семантика

**Используйте HTML так, как это было задумано.**

Используйте элементы (Иногда неверно называемые “тегами”) по назначению: заголовки для заголовков, _p_ для абзацев, _a_ для ссылок и т.д.

Это облегчает чтение, редактирование и поддержку кода.

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется -->
<div onclick="goToRecommendations();">All recommendations</div>
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
<a href="recommendations/">All recommendations</a>
```

#### Альтернатива для мультимедиа

**Всегда указывайте альтернативное содержимое для мультимедиа.**

Постарайтесь указать альтернативное содержимое для мультимедиа: например для картинок, видео или анимаций, заданных с помощью _canvas_. Для картинок это осмысленный альтернативный текст (_alt_), а для видео и аудио расшифровки текста и подписи если это возможно.

Альтернативное содержимое может помочь людям с с ограниченными возможностями. Например человеку со слабым зрением сложно понять, что на картинке если для нее не задан _@alt_. Другим людям может быть тяжело понять о чем говорится в видео или аудио записи.

(Если для картинки _alt_ избыточен, или она используется только в декоративных целях в местах, где нельзя использовать CSS, используйте пустой альтернативный текст _alt=""_)

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется -->
<img src="spreadsheet.png">
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
<img src="spreadsheet.png" alt="Spreadsheet screenshot.">
```

#### Разделение ответственности

**Разделяйте структуру, оформление и поведение.**

Держите структуру (разметка), оформление (стили) и поведение (скрипты) раздельно и постарайтесь свести взаимодействие между ними к минимуму.

Убедитесь, что документы и шаблоны содержат только HTML, и что HTML служит только для задания структуры документа. Весь код, отвечающий за оформление, перенесите в файлы стилей, а код отвечающий за поведение — в скрипты.

Старайтесь сократить их пересечения к минимуму, включая в шаблоны минимальное количество файлов стилей и скриптов.

Отделение структуры от представления и поведения помогает облегчить поддержку кода. Изменение шаблонов и HTML-документов всегда занимает больше времени чем изменение файлов стилей или скриптов.

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется -->
<!DOCTYPE html>
  <title>HTML sucks</title>
  <link rel="stylesheet" href="base.css" media="screen">
  <link rel="stylesheet" href="grid.css" media="screen">
  <link rel="stylesheet" href="print.css" media="print">
  <h1 style="font-size: 1em;">HTML Отстой</h1>
  <p>Я об этом и раньше где-то читал, но теперь точно все ясно:
    <u>HTML - полная фигня!!1</u>
  <center>Не могу поверить, что для того чтобы изменить оформление,
  нужно каждый раз все переделывать заново. </center>
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
<!DOCTYPE html>
  <title>My first CSS-only redesign</title>
  <link rel="stylesheet" href="default.css">
  <h1>Мой новый CSS дизайн</h1>
  <p>Я читал об этом и раньше, но наконец-то сделал сам:
    Использую принцип разделения ответственности и не пихаю оформление в HTML
  <p>Как круто!
```

#### Ссылки-мнемоники

**Не используйте ссылки-мнемоники.**

Нет смысла использовать ссылки-мнемоники, такие как `&mdash;`, `&rdquo;`, или `&#x263a;`, когда все команды в файлах, редакторах используют одну кодировку (UTF-8).

Единственное исключение из этого правила — служебные символы HTML (например `<` и `&`) а так же вспомогательные и “невидимые” символы (например неразрывный пробел).

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется -->
Валютный знак евро: &ldquo;&eur;&rdquo;.
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
Валютный знак евро: “€”.
```

#### Необязательные теги

**Не используйте необязательные теги. (не обязательно)**

Для уменьшения размера файлов и лучшей читаемости кода можно опускать необязательные теги. В [спецификации HTML5](https://html.spec.whatwg.org/multipage/syntax.html#syntax-tag-omission) есть список необязательных тегов.

(Может потребоваться некоторое время для того, чтобы этот подход начал использоваться повсеместно, потому что это сильно отличается от того, чему обычно учат веб-разработчиков. С точки зрения, согласованности, и простоты кода лучше всего опускать все необязательные теги, а не некоторые из них).

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется -->
<!DOCTYPE html>
  <html>
    <head>
      <title>Тратим байты - тратим деньги.</title>
    </head>
    <body>
      <p>Вот.</p>
    </body>
  </html>
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
<!DOCTYPE html>
  <title>Байты-деньги!</title>
  <p>Так-то
```

#### Атрибут `type`

**Не указывайте атрибут `type` при подключении стилей и скриптов в документ.**

Не используйте атрибут `type` при подключении стилей (кроме вариантов когда используется что-то кроме CSS) и скриптов (кроме вариантов когда это не JavaScript).

Указывать атрибут `type` в данном случае не обязательно потому что HTML5 использует `text/css` и `text/javascript` по умолчанию. Это будет работать даже в старых браузерах.

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется -->
<link rel="stylesheet" href="//www.google.com/css/maia.css"
  type="text/css">
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
<link rel="stylesheet" href="//www.google.com/css/maia.css">
```

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется -->
<script src="//www.google.com/js/gweb/analytics/autotrack.js"
  type="text/javascript"></script>
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
<script src="//www.google.com/js/gweb/analytics/autotrack.js"></script>
```

#### Атрибут `id`

**Избегайте лишних атрибутов `id`**.

Предпочтительней будет использовать атрибуты `class` для стилизации и `data` для скриптов.

Если атрибуты `id` строго обязательны, всегда включайте дефис в значение, чтобы убедиться, что оно не соответствует синтаксису идентификатора JavaScript, например, используйте `user-profile`, а не просто `profile` или `userProfile`.

Когда элемент имеет атрибут `id`, браузеры делают его доступным в [качестве именованного свойства глобального прототипа окна](https://html.spec.whatwg.org/multipage/window-object.html#named-access-on-the-window-object), что может привести к неожиданному поведению. Хотя значения атрибута `id`, содержащие дефис, по-прежнему доступны в качестве имен свойств, на них нельзя ссылаться как на глобальные переменные JavaScript.

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется: `window.userProfile` будет ссылаться на <div> элемент -->
<div id="userProfile"></div>
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется: `id` атрибут обязателен, и его значение включает дефис -->
<div aria-describedby="user-profile">
  …
  <div id="user-profile"></div>
  …
</div>
```

### Правила форматирования HTMl

#### Форматирование

**Выделяйте новую строку для каждого блочного, табличного или списочного элемента и ставьте отступы для каждого дочернего элемента.**

Независимо от стилей заданных для элемента (CSS позволяет изменить поведение элемента с помощью свойства `display`), переносите каждый блочный или табличный элемент на новую строку.

Также ставьте отступы для всех элементов вложенных в блочный или табличный элемент.

(Если у вас возникнут сложности из-за пробельных символов между списочными элементами, допускается поместить все li элементы в одну строку. Линту рекомендуется в данном случае выдавать предупреждение вместо ошибки.)

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
<blockquote>
  <p><em>Space</em>, the final frontier.</p>
</blockquote>
```

```html
<!-- ✅ Рекомендуется -->
<ul>
  <li>Маша</li>
  <li>Глаша</li>
  <li>Чебураша</li>
</ul>
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
<table>
  <thead>
    <tr>
      <th scope="col">Прибыль
      <th scope="col">Налоги
  <tbody>
    <tr>
      <td>$ 5.00
      <td>$ 4.50
</table>
```

#### Перенос строк в HTML

**Переносите длинные строки (необязательно).**

Хотя в HTML нет рекомендаций по ограничению столбцов, вы можете рассмотреть возможность переноса длинных строк, если это значительно улучшает читаемость.

При переносе строк каждая строка продолжения должна иметь отступ, чтобы отличать перенесенные атрибуты от дочерних элементов. Строки должны быть последовательно перенесены в рамках проекта, что в идеале обеспечивается средствами автоматического форматирования кода.

{/* prettier-ignore */}
```html
<button
  mat-icon-button
  color="primary"
  class="menu-button"
  (click)="openMenu()"
>
  <mat-icon>menu</mat-icon>
</button>
```

{/* prettier-ignore */}
```html
<button mat-icon-button color="primary" class="menu-button"
    (click)="openMenu()">
  <mat-icon>menu</mat-icon>
</button>
```

{/* prettier-ignore */}
```html
<button
    mat-icon-button
    color="primary"
    class="menu-button"
    (click)="openMenu()">
  <mat-icon>menu</mat-icon>
</button>
```

{/* prettier-ignore */}
```html
<button mat-icon-button
        color="primary"
        class="menu-button"
        (click)="openMenu()">
  <mat-icon>menu</mat-icon>
</button>
```

#### Кавычки в HTML

**При заключении значений атрибутов в кавычки используйте двойные кавычки.**

Используйте двойные (""), а не одинарные кавычки ('') вокруг значений атрибутов.

{/* prettier-ignore */}
```html
<!-- ❌ Не рекомендуется -->
<a class='maia-button maia-button-secondary'>Sign in</a>
```

{/* prettier-ignore */}
```html
<!-- ✅ Рекомендуется -->
<a class="maia-button maia-button-secondary">Sign in</a>
```

## CSS

### Правила оформления CSS

#### Валидность CSS

**По возможности используйте валидный CSS-код.**

Кроме случаев, где необходим браузеро-зависимый код, или ошибок валидатора, используйте валидный CSS код.

Используйте такие инструменты как [W3C CSS validator](https://jigsaw.w3.org/css-validator/) для проверки своего кода.

Валидность — это важное и при этом измеряемое качество кода. Написание валидного CSS помогает избавиться от избыточного кода и обеспечивает правильное использование таблиц стилей…

#### Идентификаторы и названия классов

**Используйте шаблонные или имеющие смысл имена классов и идентификаторы.**

Вместо использования шифров, или описания внешнего вида элемента, попробуйте в имени класса или идентификатора выразить смысл его создания, или дайте ему шаблонное имя…

рекомендуется выбирать имена, отражающие сущность класса, потому что их проще понять и, скорее всего, не понадобится менять в будущем.

Шаблонные имена — это просто вариант названия для элементов, у которых нет специального предназначения или которые не отличаются от своих братьев и сестер. Обычно они необходимы в качестве “Помощников.”

Использование функциональных или шаблонных имен уменьшает необходимость ненужных изменений в документа или шаблонах.

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется: не имеет смысла */
#yee-1901 {}

/* ❌ Не рекомендуется: описание внешнего вида */
.button-green {}
.clear {}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется: точно и по делу */
#gallery {}
#login {}
.video {}
  
/* ✅ Рекомендуется: шаблонное имя */
.aux {}
.alt {}
```

#### Названия идентификаторов и классов

**Для идентификаторов и классов используйте настолько длинные имена, насколько нужно, но настолько короткие, насколько возможно.**

Попробуйте сформулировать, что именно должен делать данный элемент, при этом будьте кратки насколько возможно.

Такое использование классов и идентификаторов вносит свой вклад в облегчение понимания и увеличение эффективности кода.

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется */
#navigation {}
.atr {}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
#nav {}
.author {}
```

#### Разделители имен классов

**Разделяйте слова в именах классов дефисом.**

Не объединяйте слова и сокращения в селекторах никакими символами (вообще никакими), кроме дефисов, для улучшения понимания и читабельности.

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется: не разделяются слова "demo" и "image" */
.demoimage {}

/* ❌ Не рекомендуется: используется нижнее подчеркивание вместо дефиса */
.error_status {}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
.video-id {}
.ads-sample {}
```

#### Префиксы

**Селекторы префиксов с префиксом, специфичным для конкретного приложения (необязательно).**

В крупных проектах, а также для кода, который внедряется в другие проекты или на внешние сайты, используйте префиксы (как пространства имен) для имен классов. Используйте короткие уникальные идентификаторы, за которыми следует тире.

Использование пространств имен помогает предотвратить конфликты именований и может упростить обслуживание, например, при операциях поиска и замены.

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
.adw-help {} /* AdWords */
.maia-note {} /* Maia */
```

#### Селекторы типа

**Избегайте использование имен классов или идентификаторов с селекторами типа (тега) элемента.**

Кроме случаев когда это не обходимо (например с классами-помощниками), не используйте названия элементов с именами классов или идентификаторами.

Это повышает [производительность](http://www.stevesouders.com/blog/2009/06/18/simplifying-css-selectors/).

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется */
ul#example {}
div.error {}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
#example {}
.error {}
```

#### ID селекторы

**Избегайте ID селекторы.**

Ожидается, что ID атрибуты будут уникальными для всей страницы, что трудно гарантировать, когда страница содержит множество компонентов, над которыми работали многие разные инженеры. Селекторам классов следует отдавать предпочтение во всех ситуациях.

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется */
#example {}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
.example {}
```

#### Сокращенные формы записи свойств

**Используйте сокращенные формы записи свойств, где возможно.**

CSS предлагает множество различных [сокращенных](https://www.w3.org/TR/CSS21/about.html#shorthand) форм записи (например `font`), которые рекомендуется использовать везде где это возможно, даже если задается только одно из значений.

Использование сокращенной записи свойств полезно для большей эффективности и лучшего понимания кода.

```css
/* ❌ Не рекомендуется */
.test {
  border-top-style: none;
  font-family: palatino, georgia, serif;
  font-size: 100%;
  line-height: 1.6;
  padding-bottom: 2em;
  padding-left: 1em;
  padding-right: 1em;
  padding-top: 0;
}
```

```css
/* ✅ Рекомендуется */
.test {
  border-top: 0;
  font: 100%/1.6 palatino, georgia, serif;
  padding: 0 1em 2em;
}
```

#### 0 и единицы измерения

**Не указывайте единицы измерения для нулевых значений.**

Не указывайте единицы измерения для нулевых значений если на это нет причины.

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется */
#example {}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
.example {
  flex: 0px; /* flex-basis необходима единица измеренеия */
  flex: 1 1 0px; /* не двусмысленно без единицы измерения, но необходимо в IE11 */
  margin: 0;
  padding: 0;
}
```

#### 0 в целой части

**Всегда ставьте 0 в целой части.**

Поставьте `0` перед значениями или длинами от -1 до 1.

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
.example {
  font-size: 0.8em;
}
```

#### Шестнадцатеричные названия цветов

**Используйте трехсимвольную шестнадцатеричную запись где это возможно.**

Трехсимвольная шестнадцатиричная запись для цветов короче и занимает меньше места.

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется */
.example {
  color: #eebbcc;
}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
.example {
  color: #ebc;
}
```

#### Использование `!important`

**Избегайте использование `!important`.**

Эти объявления нарушают естественный каскад CSS и затрудняют рассуждение о стилях и их составление. Вместо этого используйте специфику селектора для переопределения свойств.

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется */
.example {
  font-weight: bold !important;
}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
.example {
  font-weight: bold;
}
```

#### Хаки

**Избегайте использования информации о версии браузеров, или CSS "хаков" — сперва попробуйте другие способы.**

Кажется заманчивым бороться с различиями в работе разных браузеров с помощью CSS-фильтров, хаков или прочих обходных путей. Все эти подходы могут быть рассмотрены лишь в качестве последнего средства, если вы хотите получить эффективную и легко поддерживаемую кодовую базу. Проще говоря, допущение хаков и определения браузера повредит проекту в долгосрочной перспективе, так как это означает, что проект идет по пути наименьшего сопротивления. Что облегчает использование хаков и позволяет использовать их все чаще и чаще, что в результате приведет к слишком частому их использованию.

### Правила форматирования CSS

#### Упорядочивание объявлений

**Сортируйте объявления по алфавиту.**

Задавайте объявления в алфавитном порядке, чтобы получить согласованный код, с которым легко работать.

При сортировке игнорируйте браузерные префиксы. При этом, если для одного свойства используются несколько браузерных префиксов, они также должны быть отсортированы (например -moz должен быть перед --webkit)

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
.example {
  background: fuchsia;
  border: 1px solid;
  -moz-border-radius: 4px;
  -webkit-border-radius: 4px;
  border-radius: 4px;
  color: black;
  text-align: center;
  text-indent: 2em;
}
```

#### Отступы в блоках.

**Всегда ставьте отступы для содержимого блоков.**

Всегда ставьте отступы для любого [блочного содержимого](https://www.w3.org/TR/CSS21/syndata.html#block), Например для правил внутри правил или объявлений, чтобы отобразить иерархию и облегчить понимание кода.

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
@media screen, projection {
  html {
    background: #fff;
    color: #444;
  }
}
```

#### После объявлений

**Ставьте точку с запятой после каждого объявления.**

После каждого объявления ставьте точку с запятой для согласованности кода и облегчения добавления новых свойств.

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется */
.test {
  display: block;
  height: 100px
}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
.test {
  display: block;
  height: 100px;
}
```

#### После названий свойств

**Используйте пробелы после двоеточий в объявлениях.**

Всегда используйте один пробел после двоеточия (но не до) в объявлениях, для порядка в коде.

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется */
h3 {
  font-weight:bold;
}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
h3 {
  font-weight: bold;
}
```

#### Разделение блоков объявления

**Используйте пробел между последним селектором и блоком объявления.**

Всегда используйте один пробел между последним селектором и открывающей фигурной скобкой, с которой начинается [блок объявления](https://www.w3.org/TR/CSS21/syndata.html#rule-sets).

Открывающая фигурная скобка должна находиться в той же строке, что и последний селектор в данном правиле.

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется: отсутствует пробел */
.video{
  margin-top: 1em;
}

/* ❌ Не рекомендуется: необязательное разделение строки */
.video
{
  margin-top: 1em;
}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
.video {
  margin-top: 1em;
}
```

#### Отделение селектора и объявления

**Отделяйте селекторы и объявления переносом строки.**

Начинайте каждый селектор или объявление с новой строки.

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется */
a:focus, a:active {
  position: relative; top: 1px;
}

```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
h1,
h2,
h3 {
  font-weight: normal;
  line-height: 1.2;
}
```

#### Разделение правил

**Разделяйте правила переносом строки.**

Всегда ставьте перенос строки между правилами.

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
html {
  background: #fff;
}

body {
  margin: auto;
  width: 50%;
}
```

#### Кавычки в CSS

**Используйте одинарные (`''`), а не двойные (`""`) кавычки для селекторов атрибутов и значений свойств.**

Не используйте кавычки в значениях URL (`url()`).

Исключение: Если вам действительно нужно использовать правило `@charset`, используйте двойные кавычки — [одинарные кавычки недопустимы](https://www.w3.org/TR/CSS21/syndata.html#charset).

{/* prettier-ignore */}
```css
/* ❌ Не рекомендуется */
@import url("https://www.google.com/css/maia.css");

html {
  font-family: "open sans", arial, sans-serif;
}
```

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */
@import url(https://www.google.com/css/maia.css);

html {
  font-family: 'open sans', arial, sans-serif;
}
```

### Мета правила CSS

#### Группировка правил

**Группируйте правила и обозначайте группы комментарием. (не обязательно).**

По возможности объединяйте правила в группы. Обозначайте группы комментариями и разделяйте переносом строки.

{/* prettier-ignore */}
```css
/* ✅ Рекомендуется */

/* Header */

.adw-header {}

/* Footer */

.adw-footer {}

/* Gallery */

.adw-gallery {}
```

## Заключение

Будьте последовательны

Если вы редактируете код, потратьте несколько минут, чтобы разобраться в том, как он написан. Если математические операторы обособлены пробелами, делайте то же самое. Если комментарии окружены скобочками или черточками, сделайте то же со своими комментариями.

Идея этого руководства в том, чтобы создать общий словарь, который позволил бы разработчикам сконцентрироваться на том что они хотят выразить, а не на том, как.

Мы предлагаем единые правила оформления позволяющие писать код в одном стиле, но стиль кода, уже используемый в проекте, также важен.

Если ваш код будет сильно отличаться от существующего, это может сбить читающего с ритма и затруднить чтение. Постарайтесь этого избежать.
